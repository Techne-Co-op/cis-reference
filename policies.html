Row-level security — who can see and do what
Overview Schema Policies Roles Bylaws Migrations

Row-level security (RLS) is enabled on all L1 tables. Every query is filtered by who is asking. The policies follow a cite-as-you-enforce pattern: each policy name contains the bylaw section it is enforcing.

Cite-as-you-enforce: Policy names like agents_select_member_directory__1_7 embed the bylaw reference (§1.7 — Members' inspection rights). The policy name is itself documentation.

Helper Functions

Three SECURITY DEFINER functions underlie all policies. They run with elevated privilege to check role state without exposing role data directly to the calling session.

RLS helpers
has_role() boolean has_role(role text) → boolean — checks whether the calling agent holds the named role in role_assignments (active, not expired). Used in all per-role policies.
has_any_role() boolean has_any_role(roles text[]) → boolean — checks whether the calling agent holds any of the named roles. Used for multi-role policies (e.g. director OR officer).
agent_id() uuid agent_id() → uuid — returns the agents.id corresponding to auth.uid(). Maps Supabase auth to the cooperative's agent registry.
